Agile Fluency

★One Star Team

  • Stand
  • User story(Role,Feature,Value,Priority,TestCase)
  • backlogs
  • Iterations
  • Retrospectives(Well,LessWell,Action)

Benefit: Greater visibility into teams’ work

Investment: Team development and work process design.

Core Metric: Team regularly reports progress from a business value perspective.

★★Two Star Team

  • TDD
  • Diff
  • Pair Programming
  • Collective Ownership
  • Continuous Integration
  • Craftsmanship

Benefit: Low defects and high productivity.

Investment: Lowered productivity during technical skill development.

Core Metric: Team ships on market cadence.

★★★Three Star Team

  • DevOps
  • Metrics
  • Cross-functional, Self-organizing Team

Benefit: Higher value deliveries and better product decisions.

Investment: Social capital expended on incorporating business expertise into team.

Core Metric: Team provides concrete business metrics.

Continuous Integration

Why?

  • Confident
  • Reduced Risk
  • Real Progress
  • Easier to find bugs
  • Less bugs than non-CI
  • Frequent deployment ability

Practices of Continuous Integration

  • Automate the Build
    • prepare env, compile，move files，load DB from schema ability to checkout code and run build command on clean machine
  • Make Your Build Self-Testing
    • use test to make sure you are confident everything is ok
  • Everyone Commits To the Mainline Every Day
    • Small Step Less or not branch Feature Toggle
  • Every Commit Should Build the Mainline on an Integration Machine
    • github vs tfs vs svn
  • Fix Broken Builds Immediately
    • monitor check logs and location it quickly 10-20min
  • Keep the Build Fast
    • less than 10min step 1 build, unit test, functional test step 2 build, may be daily build Performance Testing or End-to-End test.if step 2 build failed，need not to fix is as soon as possible, means step 1 build need to add a new unit test or something
  • Test in a Clone of the Production Environment
    • 如果你不在生产环境做测试，所有环境差异都是风险，可能最终造成测试环境中运行正常的软件在生产环境中无法正常运行。
  • Make it Easy for Anyone to Get the Latest Executable
    • 把每次迭代中最后一个 build 放进去通常是明智的决定
  • Everyone can see what's happening
    • 持续集成中最重要的是沟通。你需要保证每个人都能轻易看到系统的状态和最新的修改。 创建自己团队的build传感器
  • Automate Deployment
    • 你必须有几个帮你将应用轻松部署到各个环境中的脚本 你还需要自动化回滚 或者蓝绿发布,parallel change 数据库的改动很麻烦，因为你不能仅仅修改 schema，你还要保证数据本身也被顺利迁移
